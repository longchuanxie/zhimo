# 开发流程规范

## 1. 总原则

所有开发任务必须严格遵循：

```text
需求领取
  ↓
需求理解
  ↓
方案设计
  ↓
编码
  ↓
编码审核
  ↓
测试
  ↓
进度更新
```

这个流程是项目强制流程，不允许跳过。

---

## 2. 流程详解

### 2.1 需求领取

输入：

- 开发任务卡；
- 需求文档；
- UI 原型；
- 数据对象设计；
- API / Service 设计。

输出：

- 明确任务负责人；
- 明确任务状态；
- 明确任务依赖；
- 明确预计交付物。

状态变化：

```text
待开发 → 开发中
```

---

### 2.2 需求理解

必须回答：

1. 用户要完成什么？
2. 为什么需要这个功能？
3. 当前 UI 原型怎么表达？
4. 涉及哪些组件？
5. 涉及哪些数据对象？
6. 涉及哪些本地服务？
7. 是否需要数据库迁移？
8. 是否涉及 AI / ContextPack？
9. 是否有空状态？
10. 是否有错误状态？

输出建议：

```text
需求理解记录
```

---

### 2.3 方案设计

必须明确：

- 页面结构；
- 组件结构；
- 数据流；
- Service 调用；
- 状态管理；
- 错误处理；
- 测试点；
- 是否影响现有模块；
- 是否需要更新文档。

输出建议：

```text
方案设计说明
```

---

### 2.4 编码

编码要求：

- 遵循 TypeScript 类型约束；
- 遵循中文 UI 规范；
- 遵循 Heroicons 图标规范；
- 遵循本地 Service 访问规范；
- 遵循数据库迁移规范；
- 遵循错误处理规范；
- 遵循安全规范。

---

### 2.5 编码审核

审核内容：

| 审核项 | 必须通过 |
|---|---|
| 是否满足需求 | 是 |
| 是否符合 UI 原型 | 是 |
| 是否中文优先 | 是 |
| 是否符合组件拆分 | 是 |
| 是否通过 Service 访问数据 | 是 |
| 是否有错误处理 | 是 |
| 是否有空状态 | 是 |
| 是否有测试 | 是 |
| 是否有安全问题 | 否 |

---

### 2.6 测试

测试类型：

- 单元测试；
- 组件测试；
- Service 测试；
- 本地数据库测试；
- 文件系统测试；
- 集成测试；
- 手动验收测试。

必须覆盖：

- 正常路径；
- 空状态；
- 错误状态；
- 边界情况；
- 数据持久化；
- 客户端重启恢复。

---

### 2.7 进度更新

更新内容：

```text
卡片编号：
当前状态：
完成内容：
测试结果：
遗留问题：
阻塞情况：
下一步：
```

状态必须从以下枚举中选择：

```text
待开发
开发中
待联调
待验收
已完成
阻塞
后置
```

---

## 3. 优化后的完整流程

在不破坏强制流程的前提下，推荐扩展为：

```text
需求领取
  ↓
需求理解
  ↓
风险识别
  ↓
方案设计
  ↓
方案评审
  ↓
编码
  ↓
自检
  ↓
编码审核
  ↓
测试
  ↓
缺陷修复
  ↓
验收
  ↓
进度更新
  ↓
文档归档
```

其中强制核心链路仍然是：

```text
需求领取 → 需求理解 → 方案设计 → 编码 → 编码审核 → 测试 → 进度更新
```
