# 项目核心流程缺陷修复与工程质量建设规划

> 规划方向：A 类（核心流程缺陷修复）+ C 类（工程质量建设）并行推进，每修一个缺陷同步补回归测试。B 类能力完整度（项目概览统计、PDF/Word 解析、模型列表选择等）延后到下一轮规划。
>
> 创建时间：2026-06-25
> 依据：代码库实际探索（Phase 1）+ 用户方向选择（选项 2）

## 一、摘要

本规划聚焦解决当前 MVP 中影响主流程可用性的 3 个核心缺陷，并同步建立工程质量的最低安全网。核心目标：

1. **修复待优化项 #5**：文档选中文本后，改写/扩写/缩写/检查来源无法触发——SelectionFloatingMenu 下发了 `pendingAgentAction` 但 AgentPanel 侧无消费逻辑（断链）。
2. **修复待优化项 #1**：设置页模型配置完成后，Agent 面板使用时仍提示"请先配置模型服务商"——AgentService 取模型配置的链路断裂。
3. **修复待优化项 #7**：Agent 面板"已采纳"未产生实质效果——`setAgentInsertText` 仅在 DocumentEditorPage 挂载时生效，非编辑页采纳无反馈。
4. **建立测试基础设施**：项目当前零测试覆盖（违反 AGENTS.md §4.6），从零搭建 vitest，并为核心 Service 与本次修复的缺陷补回归测试。
5. **弹框组件化**：用已有 Modal 组件替换散落在 9 个页面中的 21 处 `window.prompt/confirm/alert`。
6. **数据库事务补齐**：`batchExecute` 当前逐条执行无事务（db.ts:54-62 有 TODO），存在部分失败风险。

## 二、当前状态分析（基于探索事实）

### 2.1 待优化项 #5 根因

- `src/components/editor/SelectionFloatingMenu.tsx:71-85` 的 `triggerAiAction` 正确下发了三个状态：
  - `setSelectedText(text)`
  - `setAgentPanelOpen(true)`
  - `setPendingAgentAction(config)`（含 taskType + template）
- 但 Grep 在 `src/components/agent/**` 中搜索 `pendingAgentAction|selectedText` **零匹配**——AgentPanel / AgentInputArea 完全没有订阅这两个 store 字段。
- 结论：浮动菜单→store→AgentPanel 的链路在 AgentPanel 侧断开，导致打开面板后无自动填充、无自动发送。

### 2.2 待优化项 #1 根因

- `src/services/model/ModelService.ts:431-451` 的 `callModel` 依赖 `input.modelConfigId`，通过 `findConfigById` 查找配置，找不到则返回 `MODEL_NOT_CONFIGURED`（"请先配置模型服务商"）。
- 链路断裂点在 `src/services/agent/AgentService.ts:427` 附近——AgentService 发送消息时获取/传入 modelConfigId 的逻辑存在问题，导致即使设置页已配置，调用时仍取不到有效配置。
- 同样的错误提示也出现在 `ModelService.ts:439/448`、`ProjectOnboardingService.ts:78`。

### 2.3 待优化项 #7 根因

- `src/components/agent/useMessageActions.ts:57-66` 的 `adopt` 对 `rewrite`/`expand`/`summarize`/`format_text` 任务类型调用 `setAgentInsertText(message.content)`。
- `src/components/editor/Editor.tsx:104-117` 的 `useEffect` 监听 `agentInsertText`，非空时插入文档并清空。
- 问题：`Editor` 组件仅在 `src/features/document/DocumentEditorPage.tsx:151` 挂载。当用户在非文档编辑页（资料页、卡片页、大纲页等）点击 Agent 采纳时，`agentInsertText` 被设置但无消费者，用户感知"无效果"。
- 即使后续切回编辑页，`Editor` 重新挂载、`editor` 重建，store 中的 `agentInsertText` 仍为旧值，`useEffect` 会触发插入——但用户在点击瞬间无任何反馈。

### 2.4 测试基础设施现状

- 项目 `package.json` 无 vitest/jest 依赖，无 `vitest.config.*` 文件，全仓库无任何 `.test/.spec` 文件。
- 需从零搭建：依赖安装、配置文件、mock 层（Tauri SQL/FS 插件）、示例测试。

### 2.5 弹框组件化现状

- 已有 `src/components/foundation/Modal.tsx` 组件可用。
- Grep 命中 21 处 `window.prompt/confirm/alert`，分布在：
  - `src/features/document/DocumentListPage.tsx`（prompt 创建文档 + confirm 删除 + alert 失败）
  - `src/features/document/DocumentEditorPage.tsx`（alert 标题保存失败）
  - `src/features/source/SourceListPage.tsx`、`SourceDetailPage.tsx`（confirm 删除）
  - `src/features/project/ProjectListPage.tsx`、`ProjectSettingsPage.tsx`（confirm 删除 + alert 失败）
  - `src/features/outline/OutlinePage.tsx`（confirm 删除节点）

### 2.6 数据库事务现状

- `src/services/database/db.ts:54-62` 的 `batchExecute` 逐条执行 SQL，无事务包裹（代码中有 TODO 注释）。

### 阶段 3：修复待优化项 #7——Agent 采纳效果

**目标**：Agent 面板点击"采纳"后，用户能明确感知到结果已落地（文档插入 / 卡片创建 / 知识创建 / 大纲节点创建），且在非编辑页也有清晰反馈。

**根因**：`setAgentInsertText` 仅 DocumentEditorPage 挂载时生效；非编辑页采纳无反馈。

**变更文件**：

1. **`src/components/agent/useMessageActions.ts`** — 增强 `adopt` 的 rewrite/format_text 分支：
   - 当前仅 `setAgentInsertText(message.content)`，在非编辑页无效果
   - 修复方案：采纳时判断当前是否在文档编辑上下文
     - 若当前项目有"当前活动文档"（documentId），则 setAgentInsertText 触发插入

   - 决策：**采用"活动文档"机制**——在 appStore 增加 `activeDocumentId`，DocumentEditorPage 挂载时设置；adopt 时若无 activeDocumentId，则提示用户先打开文档

2. **`src/stores/appStore.ts`** — 新增 `activeDocumentId` 状态：

3. **`src/components/editor/Editor.tsx`** — 保持现有 useEffect 监听 agentInsertText 不变，确认插入逻辑正确。

## 五、假设与决策

1. **测试框架选 vitest 而非 jest**：与 vite 生态对齐，配置简单，ESM 原生支持。
2. **Tauri 插件采用 mock 而非真实调用**：单测环境无 Tauri runtime，必须 mock `@tauri-apps/api/sql`、`@tauri-apps/plugin-fs`、`@tauri-apps/plugin-dialog`。
3. **#5 采用自动发送而非仅填充**：浮动菜单指令已是完整提示词，自动发送符合"快捷动作"语义；若用户反馈需确认，可加配置开关。
4. **#7 采用"活动文档"机制而非自动创建文档**：自动创建文档可能产生大量空文档；提示用户先打开文档更可控。
5. **弹框采用命令式 hook（useConfirm）+ 全局挂载**：减少页面侵入，调用方式接近原 window.confirm，迁移成本低；失败提示统一走 toast。
6. **B 类能力完整度本轮不做**：项目概览统计、PDF/Word 解析、模型列表选择、写作目标入上下文、异步导出队列延后下一轮。
7. **待优化项 #9（上下文自动压缩）已完成**，本轮不涉及；#2（弹框组件化）纳入阶段 4；#3/#4/#6/#8 属 B 类延后。

### 阶段 5：数据库事务补齐 ✅ 已完成

**关键发现**：`@tauri-apps/plugin-sql@2.4.0` 类型定义中**没有** `executeTransaction` API（原规划假设可用，实际不可用）。改为通过 `BEGIN`/`COMMIT`/`ROLLBACK` SQL 语句在 db.ts 中实现事务，由 SQLite 引擎本身保证原子性。同步调整了 sqlMock 内存引擎与 setup.ts mock，让事务状态在测试环境下可观测。

**变更文件**：

1. **`src/services/database/db.ts`** — 修改/新增：
   - `batchExecute` 改为 `BEGIN → 多条 execute → COMMIT`，catch 中 `ROLLBACK` 并重新抛原错误
   - 新增 `runInTransaction<T>(fn)`：接受回调的事务协调器，允许 Service 层组合任意 Repository 调用，保持 Repository 的 typed API 封装
   - 放宽架构约束注释：明确"事务协调 API 允许 Service 层调用"（行业惯例，对应 Spring `@Transactional`）
2. **`src/test/fixtures/sqlMock.ts`** — 调整：
   - 删除已废弃的 `mockExecuteTransaction`（plugin-sql 2.4.0 无对应 API）
   - `mockExecute` 中识别 `BEGIN`/`COMMIT`/`ROLLBACK`，维护 `transactionSnapshot` 状态
   - `executeSql` 默认分支从"静默返回空"改为"抛错"，让无效 SQL 真的失败（事务回滚测试依赖此行为）
3. **`src/test/setup.ts`** — 移除 mockDb 中的 `executeTransaction` mock（与真实 plugin-sql 类型对齐）
4. **`src/services/database/db.test.ts`** — 测试：
   - 现有"批量执行多条语句"测试改为事务性执行
   - 新增"事务失败时回滚"测试：第二条 SQL 无效时，第一条 UPDATE 也被回滚（数据保持初始状态）
   - 新增"ROLLBACK 后可继续正常 execute"测试：验证事务状态已清理，不残留嵌套
5. **`src/services/outline/OutlineService.ts`** — `convertNodeToDocument` 改用 `runInTransaction` 包裹 `createDocument` + `updateOutlineNode`，避免孤儿文档；catch 中识别 `AppError` 形态直接透传，避免包装两层

**审查批量写入调用点**（只读排查）：
- `OutlineService.convertNodeToDocument` ✅ 已改用事务（最高优先级，孤儿文档风险）
- `AgentService.sendMessage` ⏳ 暂未改：前置段（insertMessage+insertRun+bumpStats）与后置段（insertMessage+markRunSucceeded+bumpStats）需分两段事务，中间 model 调用不能进事务。改动较大，作为技术债延后
- `ExportService.executeExport` ✅ 不需改：仅同表同行的两次 UPDATE，事务需求低，retry 可恢复

**技术债 TD-005**：
- 编号：TD-005
- 产生原因：`AgentService.sendMessage` 多表写入未事务化
- 影响范围：Agent 消息发送场景，前置段或后置段部分失败时可能出现消息记录与 run 状态不一致
- 临时方案：保持现状，依赖 model 调用前后的两次写入各自相对独立
- 后续修复建议：将前置段（消息+run+stats）与后置段（消息+run 状态+stats）分别用 `runInTransaction` 包裹
- 优先级：中
