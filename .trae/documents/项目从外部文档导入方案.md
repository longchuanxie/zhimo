# 项目从外部文档导入方案

> 本方案为 AI 原生写作项目系统新增「从已有文档开始」创建项目的入口。
> 用户选择外部文档（.md/.markdown/.txt/.docx/.pdf）→ 系统解析为结构化内容 → AI 推断项目元数据 → 用户预览编辑 → 创建项目并写入首个正文 Document。

---

## 1. 需求理解

### 用户可见行为

1. 项目列表页顶部操作栏新增第三种创建方式：「从文档创建」按钮。
2. 进入 `/projects/new-from-document` 页面，引导式三步流程：
   - **步骤 1 选文档**：点击「选择文档」按钮，弹出 Tauri 文件选择对话框，限定 `.md/.markdown/.txt/.docx/.pdf`；选中后立即调用 Rust 解析为 `StructuredDoc`，展示文件名、字数、节点数、解析进度。
   - **步骤 2 推断元数据**：调用 AI 基于 `plainText` 推断项目元数据（name/type/description/writingGoal/targetReader/targetWordCount），用户可编辑。
   - **步骤 3 确认创建**：用户确认后创建项目 + 创建首个正文 Document（标题默认「正文」或文档主标题） + 把 `StructuredDoc` 转换为 TipTap JSON 写入 Document。
3. 创建成功后跳转到 `/projects/{projectId}/documents/{documentId}` 编辑器页。

### 涉及数据对象

- `Project`：新增一条记录
- `Document`：新增一条记录，`content_json` 为 TipTap JSON
- 不涉及 Source / Card / Outline / Knowledge

### 涉及 Service

- `ProjectService.createProject`（已有）
- `DocumentService.createDocument` + 新增 `setDocumentInitialContent`
- `ProjectInferenceService`（新增）：基于文档内容 AI 推断项目元数据
- `DocumentImportService`（新增）：编排「解析→推断→创建项目→创建文档→写入正文」
- `tiptapConverters`（新增）：`StructuredDoc` → TipTap JSON

### 状态/错误/空态覆盖

- 加载状态：解析中、AI 推断中、创建中
- 错误状态：文件格式不支持、文档为空、解析失败、AI 推断失败、模型未配置、创建失败
- 空状态：未选择文档时的引导提示

## 3. 方案设计

### 3.1 数据流

```
用户点击「从文档创建」
  → /projects/new-from-document
  → [步骤1] 用户点「选择文档」
      → SourceService 风格: open() 弹对话框
      → invoke('parse_document_structured', { filePath })
      → Rust 返回 StructuredDoc
      → 前端展示文件名/字数/节点数
  → [步骤2] 自动触发 AI 推断
      → ProjectInferenceService.inferProjectFromDocument(plainText)
      → callModelDirect({...})  // 复用 ProjectOnboardingService 模式
      → safeParseJson<{name,type,description,writingGoal,targetReader,targetWordCount}>
      → 表单回填,用户可编辑
  → [步骤3] 用户确认「创建项目」
      → DocumentImportService.createProjectFromDocument(input)
        ├─ ProjectService.createProject({name,type,description,...})
        ├─ DocumentService.createDocument({projectId, title})
        ├─ DocumentService.setDocumentInitialContent({documentId, contentJson, plainText, wordCount})
        └─ 返回 {project, document}
      → navigate(`/projects/${projectId}/documents/${documentId}`)
```

### 3.2 组件拆分

```
CreateProjectFromDocumentPage  (路由组件, 编排三步)
  ├─ DocumentImportStepSelect   (步骤1: 文件选择 + 解析)
  ├─ DocumentImportStepInfer    (步骤2: AI 推断 + 表单编辑)
  └─ DocumentImportStepConfirm  (步骤3: 摘要确认 + 创建)
```

单组件控制在 250 行内；表单字段复用 `CreateProjectPage` 的 `FormField`/`TypeOption` 风格但本文件内独立实现，避免跨 feature 引入。

### 3.3 Service 调用路径

| UI 动作 | Service 方法 | Rust/DB |
|---------|-------------|---------|
| 选文件 | `DocumentImportService.pickAndParseDocument()` 内部调 `@tauri-apps/plugin-dialog.open()` + `invoke('parse_document_structured')` | Rust `parse_document_structured` |
| AI 推断 | `ProjectInferenceService.inferProjectFromDocument(plainText)` | ModelService.callModelDirect |
| 创建项目 | `ProjectService.createProject` | projectRepository.insertProject + outlineRepository.insertOutline |
| 创建文档 | `DocumentService.createDocument` | documentRepository.insertDocument |
| 写入正文 | `DocumentService.setDocumentInitialContent` | documentRepository.updateDocumentContent |

### 4.1 Rust 层（仅修复，无新增）

#### `src-tauri/src/source_parser/docx.rs`

**改什么**：修复 `extract_docx_structured`、`collect_paragraph_runs`、`collect_table_rows` 三处函数对 docx-rs 0.4.20 API 的误用。

**为什么**：当前代码无法通过 `cargo check`，6 处类型/字段错误。

**怎么改**：

```rust
// 1. style 访问(约 141 行)
// 旧:
let style_name = para.style.as_ref().map(|s| s.val.clone()).unwrap_or_default();
// 新:
let style_name = para
    .property
    .style
    .as_ref()
    .map(|s| s.val.clone())
    .unwrap_or_default();

// 2. bold/italic(约 270-271 行)
// 旧:
let bold = run.run_property.bold.unwrap_or(false);
let italic = run.run_property.italic.unwrap_or(false);
// 新:
let bold = run.run_property.bold.is_some();
let italic = run.run_property.italic.is_some();

// 3. Table 迭代(约 300-313 行 collect_table_rows)
// 旧:
for row in &table.rows {
    for cell in &row.cells {
        for content in &cell.children {
            if let docx_rs::TableCellContent::Paragraph(para) = content { ... }
        }
    }
}
// 新:
use docx_rs::{TableChild, TableRowChild, TableCellContent};
for row_child in &table.rows {
    let row = match row_child { TableChild::TableRow(r) => r, _ => continue };
    for cell_child in &row.cells {
        let cell = match cell_child { TableRowChild::TableCell(c) => c, _ => continue };
        let mut cell_runs: Vec<DocxRun> = Vec::new();
        for content in &cell.children {
            if let TableCellContent::Paragraph(para) = content {
                let mut runs = collect_paragraph_runs(para);
                cell_runs.append(&mut runs);
            }
        }
        cells.push(cell_runs);
    }
    rows.push(cells);
}
```

> 注意：`use docx_rs::{TableChild, TableRowChild, TableCellContent};` 已在文件顶部 import；若 `TableCellContent` 实际未导出，回退到 `docx_rs::TableCellContent::Paragraph(para)` 全限定写法。

/// 选择并解析文档,返回 StructuredDoc
export async function pickAndParseDocument(): Promise<ServiceResult<StructuredDoc>> {
  const selected = await open({
    multiple: false,
    filters: [{ name: '文档', extensions: SUPPORTED_IMPORT_EXTENSIONS }],
  })
  if (!selected) return err({ code: OPERATION_CANCELLED, message: '已取消选择', retryable: false })
  const filePath = typeof selected === 'string' ? selected : selected.path
  try {
    const doc = await invoke<StructuredDoc>('parse_document_structured', { filePath })
    if (!doc.nodes.length || !doc.plainText.trim()) {
      return err({ code: DOCUMENT_EMPTY_CONTENT, ... })
    }
    return ok(doc)
  } catch (error) {
    // 识别 Rust 端返回的错误码字符串
    return err(fromInvokeError(error, {
      'DOCUMENT_FORMAT_UNSUPPORTED': DOCUMENT_FORMAT_UNSUPPORTED,
      'SOURCE_OCR_REQUIRED': SOURCE_OCR_REQUIRED,
      default: DOCUMENT_IMPORT_FAILED,
    }))
  }
}

## 5. 错误处理

### Rust 端

- 文件读取失败：返回 `format!("读取文件失败: {}", e)` 字符串
- 不支持的扩展名：返回 `"DOCUMENT_FORMAT_UNSUPPORTED: .{}"` 字符串
- 扫描版 PDF：返回 `"SOURCE_OCR_REQUIRED"` 字符串
- docx 解析失败：返回 `format!("解析 docx 失败: {}", e)` 字符串

### 前端

`DocumentImportService.pickAndParseDocument` 内通过 `fromInvokeError` 工具识别错误码字符串，映射到 `ServiceError`：

| Rust 返回 | 前端错误码 |
|-----------|-----------|
| `DOCUMENT_FORMAT_UNSUPPORTED: .xxx` | `DOCUMENT_FORMAT_UNSUPPORTED` |
| `SOURCE_OCR_REQUIRED` | `SOURCE_OCR_REQUIRED` |
| 其他 | `DOCUMENT_IMPORT_FAILED` |

`ProjectInferenceService` 失败映射：

| 原因 | 错误码 |
|------|--------|
| 模型未配置 | `MODEL_NOT_CONFIGURED` |
| JSON 解析失败 | `PROJECT_INFERENCE_FAILED` |
| 模型超时 | `MODEL_TIMEOUT` |
| 其他 | `PROJECT_INFERENCE_FAILED` |

`createProjectFromDocument` 任一步失败直接返回原 ServiceResult.error，不额外包装。

## 10. 假设与决策

1. **AI 推断不走 AgentThread/ContextPack**：与 `ProjectOnboardingService` 一致，项目创建前的 AI 调用直接走 `callModelDirect`，因为此时项目还不存在，无法创建 AgentThread。符合 AGENTS.md §10「项目尚未创建时不经过 AgentThread/ContextPack 流程」。
2. **plainText 截取前 4000 字符**：避免上下文超限；剩余内容不影响元数据推断质量。
3. **创建失败不回滚**：MVP 简化策略。若项目创建成功但文档创建失败，用户可手动新建文档。记录为 TD-IMPORT-03。
4. **文档标题默认「正文」**：不从文档主标题自动提取（避免推断不准），用户可在步骤 3 修改。
5. **TipTap JSON 不依赖 tiptap 包类型**：在 `tiptapConverters/types.ts` 内自定义最小类型，避免编辑器运行时类型耦合到转换器。
6. **复用 `DocumentService.autosaveDocument` 的写入路径**：`setDocumentInitialContent` 调用同一个 `updateDocumentContent` 仓储方法，但不包装为「自动保存」语义（初始写入）。
7. **`fromInvokeError` 工具**：识别 Tauri invoke 抛出的错误，若为字符串则按错误码匹配，否则用 `fromUnknown`。该工具放在 `DocumentImportService.ts` 内部，不抽为公共 util（仅此一处使用）。
