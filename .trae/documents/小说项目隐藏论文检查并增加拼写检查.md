# 小说项目隐藏论文完整性面板并新增 AI 拼写检查

## 概要

当项目类型为 `fiction`（小说/长文）时：
1. 文档编辑页底部不再展示「论文完整性面板」（`PaperIntegrityPanel`），因为小说写作不存在引文/图表/公式等论文格式诉求。
2. 新增「拼写检查面板」（`SpellCheckPanel`），通过 AI 检测中文错别字与用词问题，作为小说项目的替代检查能力。

研究/论文（`research`）与自由写作（`free_writing`）项目保持现状，仍显示「论文完整性面板」。

## 提议变更

### 1. 新增类型：`SpellCheckIssue`

**文件**：[src/types/index.ts](file:///d:/workplace/idea/zhimo/src/types/index.ts)

在 `IntegrityIssue` 定义之后新增：

```ts
/// 拼写检查问题类型
export type SpellCheckIssueKind =
  | 'typo'      // 错别字
  | 'grammar'   // 语法/语病
  | 'usage'     // 用词不当

/// 拼写检查问题
export interface SpellCheckIssue {
  /// 错误类型
  kind: SpellCheckIssueKind
  /// 原文片段（包含问题的上下文，便于定位）
  original: string
  /// 问题描述
  description: string
  /// 建议修改后的文本
  suggestion: string | null
}
```

**为什么**：依据 AGENTS.md 3.5，必须建立统一类型，禁止在页面中临时拼装无类型对象。与 `IntegrityIssue` 并列但独立，避免污染论文完整性类型语义。

### 2. 新增拼写检查 Prompt

**文件（新增）**：`src/services/spellcheck/spellCheckPrompts.ts`

```ts
/// 拼写检查专用系统提示词
/// 用于小说/长文创作的中文错别字与用词检测
export const SPELL_CHECK_PROMPT = `你是中文写作校对助手。检查用户提供的文本中的错别字、语法错误和用词问题。

【检查范围】
1. typo：错别字（如"的/地/得"误用、同音字混淆、形近字误写）
2. grammar：语法/语病（成分残缺、语序不当、搭配不当）
3. usage：用词不当（词义混淆、褒贬误用、生造词语）

【规则】
- 只指出明确错误，不修改文风、不改写句意
- 不编造问题，对存疑处可标注但不强制
- original 字段必须是原文中真实出现的片段（用于定位）

【输出格式】
严格输出 JSON，不要任何额外说明：
{ "issues": [{ "kind": "typo|grammar|usage", "original": "原文片段", "description": "问题说明", "suggestion": "建议文本或null" }] }
若无任何问题，输出 { "issues": [] }。`
```

**为什么**：依据 AGENTS.md 3.6 与现有 `paperPrompts.ts` 模式，prompt 集中维护为常量，不散落在 Service 中。

### 3. 新增拼写检查 Service

**文件（新增）**：`src/services/spellcheck/SpellCheckService.ts`

职责：
- 通过 `findDocumentById` 获取文档
- 从 `contentJson` 提取纯文本（复用 `extractTextFromContentJson` 逻辑，提取为内部工具函数）
- 调用模型（参考 `KnowledgeExtractor.invokeExtraction` 模式：`getConfigByTask('chat')` → `callModel`）
- 解析 JSON，返回 `SpellCheckIssue[]`
- 参数校验、错误转换、统一返回 `ServiceResult`

签名：
```ts
export async function checkSpelling(
  documentId: EntityId,
): Promise<ServiceResult<SpellCheckIssue[]>>
```

错误处理：
- 文档不存在 → `err({ code: NOT_FOUND, message: '文档不存在', retryable: false })`
- 文档内容为空 → `ok([])`（空列表，不视为错误）
- 模型未配置 → 透传 `MODEL_NOT_CONFIGURED` 错误
- 模型调用失败 → `fromUnknown` 包装
- JSON 解析失败 → `err({ code: 'SPELL_CHECK_PARSE_FAILED', message: '校对结果解析失败', retryable: true })`

输入长度限制：参考 `KnowledgeExtractor.MAX_INPUT_LENGTH = 8000`，超长时截断并记录（避免 token 超限）。

**为什么**：依据 AGENTS.md 3.4，Service 层负责参数校验、错误转换、返回统一结果。参考 `KnowledgeExtractor` 的轻量调用模式而非完整 Agent 流程，因为输出是问题列表而非需采纳的正文内容，不属于 AGENTS.md 第10节的"AI 输出直接覆盖正文"场景。

## 假设与决策

1. **"主页面"= 文档编辑页**：用户确认隐藏范围为「仅完整性面板」，而 `PaperIntegrityPanel` 仅出现在 `DocumentEditorPage`，故本次变更聚焦该页面，不涉及侧边栏导航、编辑器工具栏按钮、Agent prompt。
2. **仅 fiction 隐藏，free_writing 保留论文面板**：依据用户确认。采用黑名单模式（仅 fiction 特殊处理），而非白名单（仅 research 显示）。
3. **拼写检查采用轻量 AI 调用**：参考 `KnowledgeExtractor` 模式（`callModel`），不创建 AgentRun/AgentMessage。理由：输出是问题列表而非需采纳的正文，不触发 AGENTS.md 第10节的"AI 输出覆盖正文"风险。
4. **手动触发检查**：与 `PaperIntegrityPanel` 一致，打开面板时自动检查一次 + 「重新检查」按钮手动触发。不做实时检查（避免频繁调用模型）。
5. **输入截断**：文档超长时截断前 8000 字符检查（与 `KnowledgeExtractor` 一致），MVP 阶段不做了分段检查。
6. **不新增数据库迁移**：拼写检查结果不持久化（每次实时检查），无需建表。
7. **中文 UI 文案**：面板标题「拼写检查」、按钮「重新检查」、通过态「拼写检查通过，未发现错别字或用词问题」，遵循 AGENTS.md 第7节中文优先。

## 验证步骤

### 功能验证
1. **小说项目**：打开小说项目的文档编辑页 → 底部应显示「拼写检查面板」，不显示「论文完整性面板」
2. **研究项目**：打开研究/论文项目的文档 → 底部应显示「论文完整性面板」，不显示「拼写检查面板」
3. **自由写作项目**：打开自由写作项目的文档 → 底部应显示「论文完整性面板」（保持现状）
4. **拼写检查-正常路径**：在小说文档中输入含错别字的文本（如"我德故乡"）→ 点击重新检查 → 应列出问题，original 为"我德故乡"、suggestion 为"我的故乡"
5. **拼写检查-空状态**：文档内容为空 → 面板显示通过态（无问题）
6. **拼写检查-无问题**：文档内容正确 → 面板显示"拼写检查通过"
7. **拼写检查-模型未配置**：未配置模型服务商 → 面板显示错误提示"请先在设置中配置模型服务商"
8. **拼写检查-AI 调用失败**：模拟模型调用失败 → 面板显示错误信息
9. **重新检查**：点击「重新检查」按钮 → 进入 loading 态 → 刷新结果
