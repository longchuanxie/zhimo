## TD-001 OCR 集成待后续迭代

**编号**:TD-001
**产生原因**:#2 PDF/Word 解析 阶段 3 OCR 集成尝试失败
**影响范围**:扫描版 PDF 无法自动识别,需用户手动处理
**临时方案**:扫描版 PDF 返回 SOURCE_OCR_REQUIRED 错误,UI 提示"暂不支持 OCR 识别,可将 PDF 转为文本型,或使用外部 OCR 工具处理后重新导入"
**后续修复建议**:
1. 等待 paddle-ocr-rs 更新以支持 ort 2.0.0-rc.12+(当前 0.6.1 与 ort rc.12 存在根本性 API 不兼容)
2. 或改用其他 OCR 方案:
   - `ocr-rs`(基于 MNN 后端,不依赖 ONNX Runtime)
   - `kreuzberg-paddle-ocr`(paddle-ocr-rs 的 fork,可能已修复兼容性)
   - 调用外部 OCR 工具(如 PaddleOCR-json 命令行,通过 `paddleocr-json-rs` 封装)
3. PDFium DLL 分发:需要在 tauri.conf.json 的 bundle.resources 中打包 PDFium 二进制
**优先级**:中(文本型 PDF + Word 解析已可用,扫描版 PDF 是边缘场景)
**状态**:待修复
**记录时间**:2026-06-26

### 不兼容详情

paddle-ocr-rs 0.6.1 编译失败于 ort 2.0.0-rc.12,具体错误:

1. `ort::value::Tensor::from_array` 接口变更:
   - 期望 `OwnedTensorArrayData<T>`,但 `ArrayBase<OwnedRepr<f32>, Dim<[usize; 4]>>` 未实现该 trait
   - paddle-ocr-rs 内部使用 `Tensor::from_array(input_tensors)?` 调用方式失效

2. `ort::Error` 类型加泛型参数:
   - 旧版:`ort::Error`
   - 新版:`ort::Error<SessionBuilder>`
   - paddle-ocr-rs 的 `OcrError` 只实现了 `From<ort::Error>`,未实现 `From<ort::Error<SessionBuilder>>`

3. `ort::Session.inputs` 字段私有化:
   - 旧版:可直接访问 `.inputs`
   - 新版:需调用 `.inputs()` 方法
   - paddle-ocr-rs 的 `base_net.rs:33` 直接访问字段失败

4. `Session.metadata().custom()` 返回类型变更:
   - 旧版:返回 `Option<String>`,可 `.expect()`
   - 新版:返回 `String`,无 `expect` 方法
   - paddle-ocr-rs 的 `crnn_net.rs:88` 调用失败

### 回滚保留的能力

- 文本型 PDF 解析((pdf-extract crate)✓
- Word .docx 解析(docx-rs crate)✓
- 解析结果分片入库(按页/段落切分)✓
- 状态流转(pending → parsing → parsed → ready / failed)✓
- 错误处理(SOURCE_PARSE_FAILED / SOURCE_OCR_REQUIRED / SOURCE_EMPTY_TEXT)✓

### 后续迭代恢复 OCR 的步骤

1. 确认 paddle-ocr-rs 或替代 crate 已修复 ort 兼容性
2. 在 Cargo.toml 中取消注释 OCR 依赖
3. 新建 `src-tauri/src/source_parser/model_downloader.rs`(模型下载)
4. 新建 `src-tauri/src/source_parser/ocr.rs`(pdfium 渲染 + OCR 推理)
5. 在 `mod.rs` 中取消注释模块声明,恢复 `parse_pdf` 的 OCR 兜底逻辑
6. 更新 `errors.ts` 中 SOURCE_OCR_REQUIRED 的 retryable 为 true
7. 更新测试用例 3 的 retryable 断言为 true
8. 配置 PDFium DLL 分发(tauri.conf.json bundle.resources)
9. 手工验收扫描版 PDF 导入流程
